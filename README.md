# BTD-EmisPred

BTD-EmisPred provides a reusable machine-learning workflow and lightweight web
application for near-infrared-II fluorescence emission wavelength prediction from
molecular structure and solvent information.

This public repository is intended for readers to retrain the model. The cleaned
training dataset is included, but data-derived model artifacts are intentionally
not included. In particular, the trained XGBoost model, selected-feature table,
hyperparameter-search result files, test metrics and OOF residual library must be
regenerated by the user.

## Installation

RDKit is usually most reliable through conda:

```bash
conda create -n nir2-emispred python=3.11 -y
conda activate nir2-emispred
conda install -c conda-forge rdkit -y
pip install -r requirements.txt
```

The code is CPU-oriented by default. The number of CPU threads is controlled by
`pipeline.max_cpu_threads` in the YAML config.

## Input Data

The default dataset is included at:

```text
data/data/nir2_emission_dataset.csv
```

It contains 1033 cleaned molecule-solvent records. To use a different dataset,
prepare a CSV file with at least these columns:

- `SMILES`: molecular structure string.
- `Solvent`: experimental solvent label.
- `λem (nm)`: experimental emission wavelength in nm. Use another name only if you also update `pipeline.target_col`.

The column names can be different, but then update the YAML fields listed below.

## Files To Edit Before Training

Edit `config.train.mainline.yaml` first.

Required path changes:

- `paths.raw_data_file`: path to your training CSV, for example
  `data/data/nir2_emission_dataset.csv` or `/your/path/to/training_data.csv`.
- `paths.output_dir`: directory where generated model artifacts and evaluation
  outputs will be written.
- `paths.trained_model_file`: output path for the trained XGBoost model.
- `paths.selected_features_file`: output path for the selected feature table.
- `paths.artifact_metadata_file`: output path for model metadata.

Required column-name changes:

- `pipeline.target_col`: emission wavelength column, default `λem (nm)`.
- `pipeline.smiles_col`: SMILES column.
- `pipeline.solvent_col`: solvent column.

Optional method changes:

- `pipeline.n_selected_features`: number of retained features after filtering
  and RFE.
- `pipeline.random_state`: train/test split and CV seed.
- `pipeline.test_size`: held-out test fraction.
- `pipeline.use_morgan_features`, `pipeline.use_fragment_features`,
  `pipeline.use_solvent_features`: feature blocks to enable.
- `pipeline.tune_final_xgb`: whether to run final XGBoost hyperparameter search.
- `pipeline.final_xgb_tuning_iterations`: number of randomized-search trials.
- `pipeline.max_cpu_threads`: maximum CPU threads used by one Python process.

## Workflow

### 1. Data Cleaning And Feature Construction

Code files used:

- `data/dataset.py`
- `emission_project/utils.py`
- `config.train.mainline.yaml`

What happens:

1. Invalid SMILES, missing labels and invalid solvent records are removed.
2. Duplicate SMILES records are optionally deduplicated.
3. Morgan fingerprints, RDKit fragment-count features and solvent one-hot
   features are generated.
4. The dataset is split into training and held-out test sets.
5. Feature filtering is performed only from the training set.

User edits normally needed:

- Keep `paths.raw_data_file` as the included dataset, or update it for your own CSV.
- Update `pipeline.target_col`, `pipeline.smiles_col`, and `pipeline.solvent_col` only if you replace the included dataset or rename columns.
- Keep `pipeline.use_solvent_features: true` if solvent should be learned.

### 2. Candidate Model Comparison

Code files used:

- `emission_project/__main__.py`
- `emission_project/train.py`
- `emission_project/model.py`

Run only model comparison by setting this in `config.train.mainline.yaml`:

```yaml
runtime:
  stage: compare
```

Then run:

```bash
EMISSION_CONFIG=config.train.mainline.yaml python -m emission_project
```

The candidate models and parameter grids are defined in
`emission_project/model.py`, function `build_model_specs()`. Edit that function
if you want to add, remove or rename candidate algorithms. For example, this is
where CatBoost, XGBoost, LightGBM, Random Forest, Gradient Boosting, KNN,
Kernel Ridge Regression and SVR search spaces are defined.

Main output:

- `Model_Comparison_10FoldCV.csv`

### 3. Final Feature Selection And XGBoost Training

Code files used:

- `data/dataset.py`
- `emission_project/train.py`
- `emission_project/test.py`
- `emission_project/model.py`

Set:

```yaml
runtime:
  stage: final
```

Then run:

```bash
EMISSION_CONFIG=config.train.mainline.yaml python -m emission_project
```

This generates the files required for later prediction:

- `XGB_Final_Model.json`
- `Final_Model_Selected_Features.csv`
- `Model_Artifacts_Metadata.json`
- `XGB_Test_Set_Metrics.csv`
- SHAP and feature-importance outputs, when supported by the trained model.

These files are data-derived and are therefore not included in this repository.
They will appear under `paths.output_dir` after training.

### 4. Optional OOF Residual Correction

Code file used by the web app:

- `web_app/app.py`

The web app can apply a gated nearest-neighbor correction if the training output
folder contains these two files:

- `NN_Residual_Correction_Config.json`
- `NN_Residual_Correction_Library.csv`

The residual library should be generated from out-of-fold predictions on the
training set. At minimum, it must contain:

- `SMILES`
- `Solvent`
- `oof_residual_true_minus_pred`

If these two files are absent, the web app and prediction code run the base
XGBoost model without nearest-neighbor correction.

### 5. Batch Prediction

Edit `config.predict.mainline.yaml` after training.

Required changes:

- `paths.output_dir`: the same output directory used during training.
- `paths.trained_model_file`: generated `XGB_Final_Model.json`.
- `paths.selected_features_file`: generated `Final_Model_Selected_Features.csv`.
- `paths.artifact_metadata_file`: generated `Model_Artifacts_Metadata.json`.
- `paths.prediction_file`: CSV containing new `SMILES` and `Solvent` values.
- Keep the `pipeline.*` feature settings consistent with the training config.

Then run:

```bash
EMISSION_CONFIG=config.predict.mainline.yaml python -m emission_project
```

Prediction input template:

```text
data/prediction/SMILES_solvent_template.csv
```

Required prediction columns:

- `SMILES`
- `Solvent`

### 6. Web App

Before starting the web app, make sure `config.predict.mainline.yaml` points to
the model artifacts generated from your own training data.

Start the app:

```bash
python web_app/app.py --host 0.0.0.0 --port 7860
```

Open:

```text
http://127.0.0.1:7860/home/prediction.html
```

The web app uses:

- `web_app/app.py` for the HTTP interface.
- `config.predict.mainline.yaml` for model and feature paths.
- `emission_project/infer.py` for feature alignment and prediction.
- `emission_project/utils.py` and `data/dataset.py` for SMILES, fragment and
  solvent feature generation.

## Generated Files That Are Not Tracked

The following files are intentionally excluded from this GitHub repository and
should be generated by each user from their own data:

- `outputs/**`
- `outputs/mainline_retrain/XGB_Final_Model.json`
- `outputs/mainline_retrain/Final_Model_Selected_Features.csv`
- `outputs/mainline_retrain/Model_Artifacts_Metadata.json`
- `outputs/mainline_retrain/NN_Residual_Correction_Config.json`
- `outputs/mainline_retrain/NN_Residual_Correction_Library.csv`
- `outputs/mainline_retrain/*metrics*.csv`
- `outputs/mainline_retrain/*tuning*.json`

`Final_Model_Selected_Features.csv` is required for prediction after training,
because it fixes the feature names and order used by the trained model. It is not
included here because it is specific to the training data and feature-selection
run.
