# Model Card: BTD-EmisPred

## Model Overview

BTD-EmisPred is a machine-learning workflow and lightweight web application for
predicting near-infrared-II (NIR-II) fluorescence emission wavelength from
molecular structure and solvent information.

The public repository is designed for reproducible retraining. It includes the
cleaned dataset and training/prediction code, while data-derived model artifacts
are intentionally excluded and must be regenerated by users.

## Intended Use

BTD-EmisPred is intended for research-stage molecular screening and analysis.
Typical use cases include:

- estimating emission wavelength from one SMILES string and one solvent label;
- comparing candidate molecules before synthesis or optical measurement;
- retraining the workflow on updated molecule-solvent datasets;
- generating model-comparison, feature-selection and explainability outputs for
  manuscript or supplementary analysis.

The model is not intended to replace experimental optical characterization.
Predictions should be treated as decision-support estimates rather than measured
values.

## Input And Output

### Inputs

Each prediction requires:

- `SMILES`: molecular structure string;
- `Solvent`: normalized experimental solvent label.

The training workflow converts these fields into:

- Morgan fingerprint features;
- RDKit fragment-count features;
- solvent one-hot encoding features.

### Output

The primary output is predicted emission wavelength in nanometers:

- `lambda_em` in nm.

When optional nearest-neighbor residual-correction artifacts are present, the web
app can also report the base XGBoost prediction, the correction term,
neighbor-similarity evidence and the final corrected prediction.

## Data

The included cleaned dataset is:

```text
data/data/nir2_emission_dataset.csv
```

It contains 1033 cleaned molecule-solvent records with molecular structure,
solvent and experimental emission wavelength labels. The dataset was prepared
from a larger source table by removing invalid or unusable records, including
records with invalid SMILES, missing emission labels, unsupported or blank
solvent records, and duplicate molecule-solvent entries according to the project
cleaning rules.

Users who replace the dataset should provide at least:

- a SMILES column;
- a solvent column;
- an emission wavelength target column in nm.

The corresponding column names must be updated in `config.train.mainline.yaml`.

## Feature Construction

The default feature representation concatenates three feature blocks:

1. Morgan fingerprints for molecular topology and local substructure patterns.
2. RDKit fragment-count features for interpretable structural and
   functional-group information.
3. Solvent one-hot features for explicit solvent-environment encoding.

Feature generation is implemented mainly in:

- `data/dataset.py`
- `emission_project/utils.py`

Feature settings are controlled by `config.train.mainline.yaml`.

## Feature Selection

Feature selection is performed only on the training subset after the train/test
split. The intended workflow avoids selecting features from the held-out test
set.

The main feature-selection steps are:

1. variance filtering;
2. correlation filtering;
3. recursive feature elimination (RFE);
4. saving the final selected feature names and order for later prediction.

The generated selected-feature file is data-derived and is therefore not tracked
by Git:

```text
outputs/mainline_retrain/Final_Model_Selected_Features.csv
```

Users must regenerate it during training. Prediction must use the same saved
feature names and order as the trained model.

## Model Selection And Training

The repository supports candidate model comparison using 10-fold
cross-validation on the training subset. Candidate regressors include:

- CatBoost;
- XGBoost;
- LightGBM;
- Random Forest;
- Gradient Boosting Regression;
- K-Nearest Neighbors;
- Kernel Ridge Regression;
- Support Vector Regression.

Candidate models and search spaces are defined in `emission_project/model.py`.

The mainline workflow selects XGBoost as the final base model. In the final
training stage, the selected feature set and optimized hyperparameters are used
to retrain the XGBoost model on the full training subset before held-out test
evaluation.

The generated trained model is data-derived and is not tracked by Git:

```text
outputs/mainline_retrain/XGB_Final_Model.json
```

## Usage

Install dependencies, update paths in `config.train.mainline.yaml`, and run:

```bash
EMISSION_CONFIG=config.train.mainline.yaml python -m emission_project
```

After training, update `config.predict.mainline.yaml` to point to the generated
model artifacts. Batch prediction can then be run with:

```bash
EMISSION_CONFIG=config.predict.mainline.yaml python -m emission_project
```

The web app can be started with:

```bash
python web_app/app.py --host 0.0.0.0 --port 7860
```

## Runtime Environment

The repository is Python-based and CPU-oriented by default. Recommended setup:

- Python 3.11;
- RDKit installed through conda-forge;
- dependencies installed from `requirements.txt`.

The number of CPU threads used by one Python process is controlled by
`pipeline.max_cpu_threads`.

## Data Leakage And Reproducibility Notes

To reduce leakage risk:

- split the dataset before feature selection;
- perform variance filtering, correlation filtering and RFE only on the training
  subset;
- reserve the held-out test subset for final evaluation;
- build optional OOF residual libraries from training-set out-of-fold
  predictions, not from held-out test labels;
- do not tune model choices or correction parameters using external validation
  molecules that will later be presented as independent evidence.

For publication-quality reporting, preserve the configuration file, random seed,
selected-feature table, trained model, evaluation outputs and prediction inputs
associated with each result.
